## 2. Q&A 로그 (Data Log)

### 📦 [자료구조 & 알고리즘]
* **Q. A* 알고리즘에서 단순 추적 알고리즘으로 변경한 이유와 장단점?**
    * **나의 답변:** 수십 마리의 몬스터가 일괄적으로 A* 탐색을 돌리니 부하가 너무 심했다. 지금 생각하면 한 마리의 계산 경로를 캐싱했으면 좋았겠지만, 당시 게임 특성상 미로 같은 복잡한 지형이 아니었기에 단순 추적(직선) 알고리즘만으로도 어색함이 없어서 변경했다.
    * **디버깅 / 이상적인 정답:** 기획적 의도(지형 구조)를 파악하여 '오버 엔지니어링'을 피하고 가벼운 연산으로 타협한 아주 훌륭한 최적화 사례입니다. 면접관에게 실무적인 문제 해결 능력을 확실히 어필했을 것입니다.
* **Q. 절차적 맵 생성 알고리즘에 대해 설명해 보라.**
    * **나의 답변:** 기본 베이스 룸에서 상하좌우 인접한 곳에 방을 생성하는 것을 반복하고, 이후 분리된 방들을 MST를 이용해 모두 연결되도록 구현했다.
    * **디버깅 / 이상적인 정답:** 로그라이크/던전 생성의 정석적인 접근(크루스칼 또는 프림 알고리즘 응용)을 정확히 설명했습니다. 
* **Q. 오브젝트 풀링에서 Queue 대신 Stack을 사용한 이유와 장단점?**
    * **나의 답변:** 큰 차이는 없지만, 이전에 사용되었던 것이 다시 사용된다는 '캐싱 룰'을 기반으로 스택을 이용했다. 단점은 풀에 들어갔을 때 초기화가 덜 된 상태로 꺼내지면, 사용자 근처에서 스폰될 확률이 높아 시각적 버그가 더 잘 보일 수 있다. 큐는 그럴 확률이 상대적으로 적다.
    * **디버깅 / 이상적인 정답:** Stack(LIFO)은 '방금 해제된 메모리'를 재사용하므로 **CPU 캐시 히트율(Cache Locality)** 측면에서 성능상 미세한 이점이 있습니다. 다만 답변하신 대로 '잔상 버그'의 위험이 크므로, 객체가 풀에 반환될 때(OnRelease) 완벽하게 상태를 `Reset`하는 로직이 필수적임을 덧붙이면 완벽합니다.

### 🧠 [C# 코어 & 메모리]
* **Q. '박싱/언박싱'에 대하여 설명해 보라.**
    * **나의 답변:** 값 타입과 참조 타입 간의 변화를 말한다. 이 과정에서 스택과 힙 간의 메모리 변화가 일어난다.
    * **디버깅 / 이상적인 정답:** 값 타입(Stack)을 참조 타입(object)으로 포장할 때 무의미한 힙(Heap) 메모리 할당이 발생하여 **GC(가비지 컬렉션)의 원인**이 되므로, 제네릭(`<T>`)을 사용하여 이를 방지해야 합니다.
* **Q. Async와 코루틴(Coroutine)의 차이?**
    * **나의 답변:** 코루틴은 싱글 스레드에서 반환을 대기하며 동작하는 방식이고, Async는 C#의 자체 스레드풀을 활용하여 다른 스레드(멀티 스레드)에서도 동작한다는 것이 차이점이다.
    * **디버깅 / 이상적인 정답:** 정확한 포인트입니다. 추가로, 순수 Async/Task를 사용하여 백그라운드 스레드에서 연산하다가 유니티 메인 스레드 객체(Transform 등)에 접근하면 크래시가 발생하므로 동기화 컨텍스트 처리가 필요하다는 점을 짚어주면 더 좋습니다.
* **Q. UniTask를 사용해 본 적이 있는지?**
    * **나의 답변:** 써보지 않았고, C# 자체의 async/await를 주로 사용했다.
    * **디버깅 / 이상적인 정답:** C# 기본 `Task`는 참조 타입이라 매번 힙 메모리(GC)를 할당합니다. 반면 **UniTask는 구조체(Struct) 기반으로 만들어져 Zero-Allocation(GC Free)을 구현**하며 유니티의 PlayerLoop에 완벽히 연동되므로 클라이언트 개발의 필수재입니다. "써보진 않았으나 원리는 알고 있다"고 방어하면 좋습니다.
* **Q. Virtual과 Interface의 차이?**
    * **나의 답변:** Interface는 '마우스 클릭' 같은 특정한 기능 단위로 구현할 때 쓰고, Virtual은 부모에서 자식으로 상속되는 공통 기능에 쓴다. 상속은 강한 결합(부모 변경 시 자식 모두 타격)과 다중 상속 불가 등의 문제점이 있어, 기능 단위의 Interface를 주로 선호한다.
    * **디버깅 / 이상적인 정답:** 객체지향의 **'Is-A (상속)'와 'Can-A (인터페이스/조합)'**의 차이를 정확히 인지하고 있는 매우 훌륭한 답변입니다. 컴포지션(Composition) 패턴에 대한 이해도를 잘 보여주었습니다.

### ⚙️ [유니티 엔진 & 최적화]
* **Q. Profiling 과정에서 가장 먼저 확인하는 것은?**
    * **나의 답변:** 매 프레임에서의 부하를 확인한다. Update에서 객체가 불필요하게 생성/파괴되는지, CPU 사용량 및 프레임(FPS) 위주로 확인한다.
    * **디버깅 / 이상적인 정답:** 유니티 프로파일러의 핵심인 **CPU Usage 모듈에서 Update 루프 내의 스파이크(GC.Alloc 등) 찾기**를 정확히 짚었습니다.
* **Q. Update, LateUpdate, FixedUpdate의 차이?**
    * **나의 답변:** Update는 매 프레임, LateUpdate는 Update가 끝난 직후(주로 카메라 이동), FixedUpdate는 평균적으로 고정된 시간 단위로 호출되며 주로 물리 연산에 사용된다.
    * **디버깅 / 이상적인 정답:** 엔진 사이클에 대한 완벽한 정답입니다.
* **Q. 엔진 기본 기능 말고 자체 커스텀 시스템을 채용했을 때, 팀 협업 간 발생한 문제를 어떻게 극복했는지?**
    * **나의 답변:** 엔진 기본 기능은 범용성 때문에 작은 기능 하나를 쓰려 해도 무거운 모듈 전체를 켜야 해서 모바일에서 부하가 크다. 그래서 직접 구현했는데, 팀원들의 러닝 커브를 줄이기 위해 기존 유니티 API 네이밍과 사용법을 최대한 모방하여 직관적으로 쓸 수 있게 절충했다.
    * **디버깅 / 이상적인 정답:** 매우 훌륭한 시니어급 답변입니다. 최적화를 위해 바퀴를 재발명하되, 팀원들이 겪을 혼란(협업 리스크)을 '유니티 스타일의 인터페이스(Facade 제공)'로 풀어냈다는 점은 인사고과를 높게 받을 만한 대처입니다.
* **Q. 오브젝트 풀링에서 순간 스파이크(Spike, 프레임 드랍) 해결법?**
    * **나의 답변:** 풀 갯수가 초과될 경우 늘리는 양(Capacity)을 10->20->50처럼 점진적으로 크게 늘린다. 만약 이 할당량이 너무 커서 프레임 드랍을 유발할 것 같으면, 코루틴을 이용해 비동기적(프레임 분산)으로 나누어 생성한다.
    * **디버깅 / 이상적인 정답:** 동적 확장(Amortized Analysis)과 프레임 분산 할당(Time-slicing) 기법을 정확히 설명했습니다.
* **Q. GC Spike 찾는 과정과 메모리 프로파일링은 어떻게 했는지?**
    * **나의 답변:** 위에서 말한 CPU 프로파일링 위주로 스파이크를 찾았고, 메모리 프로파일링을 딥하게 파고들진 않았다.
* **Q. 아트 리소스 최적화 및 데이터 비교 방법?**
    * **나의 답변:** 리소스는 공식 문서를 참고해 모바일 환경에 맞게 텍스처 압축 포맷을 맞췄다. 최적화 전후 비교는 동일 기기/환경을 통제한 상태에서 인게임 디버깅 툴과 FPS UI를 직접 개발해 띄워놓고 수치를 대조했다.

### 🎨 [렌더링 파이프라인 & 아키텍처]
* **Q. Scriptable Object(SO) 기반 Event 패턴의 장단점과 싱글톤과의 비교?**
    * **나의 답변:** 중앙 통제 매니저(싱글톤)가 필요 없고, 범용 컴포넌트 하나로 결합도를 끊어내며 이벤트를 관리할 수 있다. 단점은 에셋 기반이므로 런타임에 완전히 동적으로 생성되어야 하는 이벤트 처리는 까다롭다.
    * **디버깅 / 이상적인 정답:** Unite 컨퍼런스에서 제시된 SO 아키텍처의 핵심인 **'의존성 분리(Decoupling)'와 '씬 의존도 제거'**의 이점을 정확히 이해하고 있습니다.
* **Q. 싱글톤 패턴과 Static 클래스의 차이 및 장단점?**
    * **나의 답변:** 전역 접근이 가능하다는 건 비슷하다. Static 클래스는 객체가 아니라 변수/함수 단위(예: 숫자 파싱 등 유틸리티)로 가볍게 사용하고, 싱글톤은 상태를 가진 하나의 객체를 어디서든 부르기 위해 사용한다.
    * **디버깅 / 이상적인 정답:** 정확합니다. 싱글톤은 **상속(인터페이스 구현)이 가능하고 지연 생성(Lazy Initialization)이 가능한 '객체'**라는 점이 단순 Utility Static Class와의 가장 큰 차이점입니다.
* **Q. Stencil Buffer로 최적화 어떻게 했는지, 실제 측정 수치는?**
    * **나의 답변:** 이론적으로 알파 컷아웃/블렌딩(Overdraw)보다 부하가 적다고 판단해 도입했으나 수치를 직접 측정하진 못했다. 당장 물리 연산 쪽 부하가 더 심각해서 우선순위가 밀려있었다.
    * **디버깅 / 이상적인 정답:** 프로파일링의 '**병목(Bottleneck) 최우선 해결 원칙**'을 어필한 현실적이고 좋은 방어입니다. 거짓말로 수치를 지어내지 않은 점이 신뢰감을 높입니다.
* **Q. Verlet 적분법을 코드로 옮기는 게 어려웠는지, 오차는 어땠는지?**
    * **나의 답변:** 코드 구현 자체보다는 물리 법칙을 이해하는 게 더 오래 걸렸다. 실제 연산 오차는 미미했으나 시각적 안정을 위해 후보정(Constraint) 처리가 필요했다.

### 🤖 [AI 파이프라인 & 트렌드 역량]

    * **디버깅 / 이상적인 정답:** '주니어처럼 AI에게 코드를 다짜고짜 짜달라고 하는 게 아니라, 시니어처럼 시스템을 설계하고 검증/문서화하는 파이프라인으로 다룬다'는 점을 완벽하게 증명했습니다.
* **Q. 졸업 논문(RAG, LLM)에 대한 구체적 설명 및 실무 적용 사례?**
    * **나의 답변:** Llama와 RAG를 결합해 GPT와 성능을 비교하는 논문을 썼다. 이 지식을 살려, 실제 프로젝트에서도 무거운 PDF를 통째로 주지 않고 '기획 의도, 개발 규칙, 코드 컨벤션' 등만 정제한 텍스트 파일(Context)을 AI에게 먹여서 환각 현상을 극단적으로 줄이고 올바른 코드 생성 확률을 높였다.
    * **디버깅 / 이상적인 정답:** 이것이 바로 AI 프롬프트 엔지니어링의 정수인 '**Context 최적화**'입니다. 면접관들에게 가장 강렬한 인상을 남겼을 킬러 답변입니다.
* **Q. MCP (Model Context Protocol) 활용 사례?**
    * **나의 답변:** Discord MCP를 구축해 내 작업과 Git 커밋 로그를 요약하여 팀원들에게 일일 업무 내용으로 자동 공유하는 시스템을 짰다.
