## 2. Q&A 로그 (Data Log)

### 📦 [자료구조 & 알고리즘]
* **Q. List와 Array의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **Array**는 선언 시 크기가 고정되며 연속된 메모리 공간을 할당받아 검색 속도가 매우 빠릅니다. **List**는 내부적으로 배열을 사용하지만 크기가 동적으로 변하며, 용량을 초과하면 내부 배열 크기를 2배로 늘리고 복사하는 작업이 발생합니다.
* **Q. Stack과 Queue의 차이 및 실 게임 사용 사례?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** Stack은 LIFO(후입선출) 구조로 **UI 팝업 매니저**(가장 마지막에 띄운 창을 먼저 닫음)나 무르기(Undo) 시스템에 사용됩니다. Queue는 FIFO(선입선출) 구조로 **네트워크 패킷 처리**, 스킬 예약 시스템, 이벤트 시스템 등에 사용됩니다.
* **Q. 사다리게임이 항상 1:1 대응인 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 사다리 타기의 가로선은 인접한 두 세로선의 위치를 1:1로 맞바꾸는 **'스왑 연산'입니다. 정보의 소실이나 병합 없이 위치만 치환하는 독립적인 연산들의 연속이기 때문에, 시작점과 도착점은 반드시 1:1로 매핑됩니다.
* **Q. A* 휴리스틱의 알고리즘 구현과 이 결과가 실제 최단거리인지 아닌지 (아니라면 이유)?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** A*는 $f(n) = g(n) + h(n)$ 공식을 사용합니다. 여기서 $h(n)$(휴리스틱, 예상 비용)이 실제 목적지까지의 비용보다 **항상 작거나 같을 때(Admissible, 허용 가능할 때)만 완벽한 최단 거리를 보장**합니다. 만약 휴리스틱을 과대평가(Overestimate)하게 설정하면, 최단 거리를 보장하지 못하는 대신 연산 속도가 훨씬 빠른 탐색(Greedy)을 하게 됩니다.
* **Q. Quick Sort와 BST의 각각 사용 사례와 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **Quick Sort**는 데이터 삽입/삭제가 없고 정렬이 한 번만 필요한 고정된 배열(예: 결과창 랭킹 정렬)에 유리합니다(제자리 정렬, 메모리 이점). **BST**는 게임 플레이 중 데이터의 삽입/삭제/탐색이 실시간으로 빈번하게 일어나는 구조(예: 실시간 매치메이킹 대기열)에 유리합니다.

### 🧠 [C# 코어 & 메모리]
* **Q. struct와 class의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **struct**는 값 타입(Value Type)으로 Stack 영역에 할당되어 GC(가비지 컬렉터)의 영향을 받지 않지만, 전달 시 데이터가 통째로 복사됩니다. **class**는 참조 타입(Reference Type)으로 Heap 영역에 할당되어 GC의 관리 대상이 되며, 전달 시 메모리 주소만 복사됩니다.
* **Q. 박싱과 언박싱의 개념, 문제점과 방지대책?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 박싱(Boxing)은 값 타입(Stack)을 참조 타입(object)으로 변환하여 Heap에 할당하는 과정입니다. 이때 무의미한 힙 메모리 할당과 GC 유발이라는 심각한 오버헤드가 발생합니다. 이를 방지하기 위해 제네릭(Generic, `<T>`)을 사용하여 타입 캐스팅 없이 컴파일 타임에 타입을 확정 지어야 합니다.
* **Q. overried와 new의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** `override`는 부모의 가상(virtual) 메서드를 재정의하여 **완벽한 다형성**을 구현합니다(업캐스팅된 상태라도 자식의 메서드가 호출됨). `new`는 부모 메서드를 단순히 숨기는(Shadowing) 것으로, 참조하고 있는 타입(부모냐 자식이냐)에 따라 호출되는 함수가 달라집니다.
* **Q. GetHashCode() 함수의 역할과 override 하는 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** `Dictionary`나 `HashSet` 같은 해시 기반 컬렉션에서 객체를 식별하는 정수 값을 반환합니다. 직접 만든 클래스를 Key로 사용할 때 내용물이 같으면 같은 Key로 인식하게 하려면, `Equals()`와 함께 `GetHashCode()`를 동일한 로직으로 **반드시 재정의**해야 합니다.
* **Q. 아래 함수가 잘못된 이유와 올바른 사용법?**
    ```csharp
    public T A<T>()
    {
        return new T();
    }
    ```
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 컴파일 에러가 발생합니다. 제네릭 타입 `T`가 기본 생성자(매개변수가 없는 생성자)를 가지고 있는지 컴파일러가 보장할 수 없기 때문입니다. 올바른 사용법은 제약 조건을 추가하는 것입니다: `public T A<T>() where T : new() { return new T(); }`
* **Q. async에서 Task를 반환 안 하고 void(null)를 반환할 때의 문제점?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** `async void`는 'Fire and Forget' 방식입니다. 호출자(Caller) 측에서 해당 비동기 작업이 끝났는지 `await`로 기다릴 수 없고, 가장 치명적인 것은 **내부에서 예외(Exception)가 발생했을 때 외부의 try-catch로 잡을 수 없어 애플리케이션 전체가 크래시(Crash)** 될 수 있습니다. (이벤트 핸들러를 제외하고는 항상 `async Task`를 반환해야 합니다.)

### ⚙️ [유니티 엔진 & 최적화]
* **Q. Sprite Atlas의 개념과 오히려 부하를 일으키는 경우?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 여러 개의 스프라이트를 하나의 큰 텍스처로 묶어 **Draw Call(배칭)을 줄여주는 최적화 기법**입니다. 오히려 부하를 일으키는 경우는, 거대한 아틀라스 파일 중 아주 작은 이미지 단 하나만 씬에서 사용하더라도 아틀라스 전체(VRAM)가 메모리에 강제로 로드되어 **메모리 낭비**가 극심해질 때입니다.
* **Q. 오픈월드에서 오브젝트의 거리를 알 수 있을 때 최적화 방법?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 1. **LOD (Level of Detail):** 거리에 따라 로우 폴리곤/저해상도 텍스처로 스왑. 2. **Culling:** 절두체 컬링(Frustum), 오클루전 컬링(Occlusion) 활용. 3. **Tick Rate 조절:** 멀리 있는 적의 AI 연산 주기(Update)를 1초에 1번으로 낮추거나 아예 비활성화.
* **Q. Dynamic Batching이 깨지는 사례들?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 동적 배칭은 조건이 매우 까다롭습니다. 버텍스 수가 너무 많을 때(보통 300개 이상), 서로 **다른 머티리얼**을 사용할 때, 다중 패스(Multi-pass) 셰이더를 사용할 때, 트랜스폼의 **스케일(Scale)에 음수 값**이 포함되어 있을 때 배칭이 깨집니다.
* **Q. 오브젝트 풀링의 개념과 장점?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 런타임에 Instantiate/Destroy를 반복하지 않고, 시작할 때 미리 객체들을 생성해(Pool) 두고 껐다 켰다(Active/Deactive) 하며 재사용하는 기법입니다. 객체 생성/파괴의 CPU 오버헤드를 줄이고, 가장 핵심적으로는 힙 메모리 단편화를 막아 **GC(가비지 컬렉터) 스파이크 프레임 드랍을 방지**합니다.
* **Q. ActiveSelf와 ActiveInHierachy의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** `activeSelf`는 자기 자신의 인스펙터 체크박스 상태(로컬 상태)만 반환합니다. `activeInHierarchy`는 부모 오브젝트가 비활성화되어 자신이 실제로 화면에 안 보이는 상태인지(글로벌 상태)까지 확인하여 반환합니다.
* **Q. Destroy와 DestroyImmediate의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** `Destroy`는 호출 즉시 파괴하지 않고, 해당 프레임의 Update 루프가 끝난 뒤 안전하게 메모리에서 해제합니다. `DestroyImmediate`는 호출 즉시 객체를 날려버리며, 주로 에디터 확장 스크립트에서만 사용해야 하고 런타임 중 컬렉션 순회 시 사용하면 치명적인 버그를 냅니다.
* **Q. 물리연산에 FixedDeltaTime을 사용하는 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 렌더링 프레임(Update)은 기기 사양에 따라 가변적입니다. 만약 물리 연산을 가변 프레임에 맡기면, 프레임 드랍이 발생할 때 캐릭터가 벽을 뚫고 지나가는 등(터널링)의 문제가 생깁니다. 물리 엔진의 **결정론적이고 일관된 시뮬레이션**을 보장하기 위해 고정된 시간 간격(FixedUpdate)을 사용합니다.
* **Q. 아래 함수의 출력 결과는?**
    ```csharp
    IEnumerator Foo()
    {
        Destroy(obj);
        yield return null;
        Debug.Log(obj);
    }
    ```
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **null (정확히는 유니티의 가짜 null 객체 표시)**이 출력됩니다. `Destroy(obj)`가 프레임 끝에 실행되는데, `yield return null`을 통해 다음 프레임으로 로직이 넘어갔기 때문에 해당 시점에서는 이미 객체가 파괴되어 있습니다.

### 🎨 [렌더링 파이프라인 & 3D 수학]
* **Q. Defered Rendering과 Foward Rendering의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **Forward**는 폴리곤마다 모든 조명 연산을 즉시 수행하여 가볍고 반투명 처리가 좋지만, 조명이 많아질수록 기하급수적으로 무거워집니다($O(Objects \times Lights)$). **Deferred**는 화면에 보이는 픽셀의 정보(G-Buffer)를 먼저 다 그린 후, 화면 픽셀 기준으로 조명을 한 번에 계산합니다($O(Pixels \times Lights)$). 빛이 아무리 많아도 성능 저하가 적지만, 메모리 대역폭을 많이 먹고 반투명(Alpha) 처리가 매우 어렵습니다.
* **Q. Alpha Blending과 Alpha Test의 차이?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **Blending**은 뒤에 있는 색상과 섞어 반투명을 표현하며, Z-Buffer에 기록하지 못해 그리는 순서(Sorting)가 매우 중요하고 오버드로우 비용이 큽니다. **Alpha Test(Cutout)**는 픽셀의 알파값이 특정 임계값(Threshold) 이하면 아예 렌더링을 폐기(Discard)합니다. Z-Buffer를 사용할 수 있어 정렬이 필요 없지만, 테두리가 계단 현상처럼 날카롭게 깨집니다.
* **Q. 렌더링 우선순위 (Render Queue, Z-Order, Order in Layer, Sorting Layer)?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 유니티의 렌더링 우선순위는 **Camera Depth -> Sorting Layer -> Order in Layer -> Render Queue -> Z-Buffer(거리)** 순으로 결정됩니다. (앞의 조건이 같을 때만 뒤의 조건을 판별합니다.)
* **Q. 내적 공식과 성질, 게임 실제 사용?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 공식: $A \cdot B = |A||B|\cos(\theta)$. 두 벡터의 정규화된 내적 값은 코사인 값(-1 ~ 1)이 됩니다. 게임에서는 **적이나 플레이어가 내 앞(>0)에 있는지 뒤(<0)에 있는지 판별**할 때, 시야각(FOV) 내에 들어왔는지 체크할 때, 광원과 표면 노멀의 조명 계산(Lambertian)에 주로 사용합니다.
* **Q. 오일러 말고 쿼터니언을 이용하는 이유와 실 사례?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 오일러 각(Euler Angles)은 X, Y, Z축을 순차적으로 회전시키기 때문에 두 축이 겹쳐버리면 한 축의 회전 자유도를 잃어버리는 **짐벌락(Gimbal Lock)** 현상이 발생합니다. 쿼터니언(4원수)은 축과 각도 기반으로 한 번에 회전하여 이를 완벽히 방지하며, 유도탄의 궤적처럼 두 회전 상태를 부드럽게 보간하는 **Slerp(구면 선형 보간)**에 필수적으로 사용됩니다.

### 🏛️ [아키텍처 & 패턴 & 기타]
* **Q. 상속 대신 조합(Composition)을 사용하는 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 상속을 깊게 사용하면 부모 클래스의 변경이 모든 자식에게 영향을 미치는 '강한 결합(Fragile Base Class)'이 발생하고, 다중 상속의 다이아몬드 문제가 발생합니다. 조합(유니티의 컴포넌트 패턴)은 필요한 기능만 레고 블록처럼 조립하여 유연성과 확장성을 극대화합니다.
* **Q. MVC, MVP, MVVM의 개념과 요소들의 역할?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 
        * **MVC:** Model(데이터), View(UI), Controller(입력 및 흐름 제어). View와 Model의 의존성이 존재함.
        * **MVP:** Presenter가 View와 Model 사이의 중재자 역할을 완벽히 하여 View와 Model을 분리(인터페이스 기반).
        * **MVVM:** ViewModel을 사용. View는 ViewModel을 관찰(Data Binding)하며, 데이터가 바뀌면 UI가 자동으로 반응함(옵저버 패턴). UGUI나 UI Toolkit에서 사용하기 좋음.
* **Q. View와 Business Logic을 분리하는 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 관심사 분리(Separation of Concerns). 스파게티 코드를 방지하고, UI가 완전히 변경되거나 다른 엔진으로 포팅되더라도 핵심 비즈니스 로직(데이터/서버 통신/전투 공식)은 그대로 재사용하고 유닛 테스트를 수행할 수 있기 때문입니다.
* **Q. 의존성 주입(DI)의 개념과 장점?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** 객체 내부에서 직접 하위 객체(`new`)를 생성하지 않고, 외부(생성자나 인터페이스)에서 객체를 주입받는 기법입니다. 결합도를 낮추고 유연성을 높이며, 가장 큰 장점은 테스트 시 가짜 객체(Mock)를 주입하여 쉽게 **단위 테스트(Unit Test)**를 할 수 있다는 점입니다.
* **Q. Behavior Tree의 개념과 동작방법?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** AI의 행동을 트리 구조(루트 -> 자식 노드)로 설계하는 방식입니다. FSM(유한 상태 기계)이 상태가 많아지면 전이(Transition)가 꼬이는 단점을 극복하기 위해 나옵니다. Selector(하나라도 성공하면 성공), Sequence(순차적으로 실행), Decorator(조건문) 등의 노드를 통해 매 틱(Tick)마다 트리를 순회하며 행동을 결정합니다. 모듈화가 뛰어나 확장성이 좋습니다.
* **Q. 인앱 결제시 게임 서버에서 결제 영수증을 확인하는 이유?**
    * **나의 답변:** 
    * **디버깅 / 이상적인 정답:** **클라이언트의 무결성을 신뢰할 수 없기 때문입니다.** 악의적인 유저나 메모리 해킹(Lucky Patcher 등)을 통해 클라이언트가 가짜 "결제 성공" 신호를 서버로 보낼 수 있습니다. 따라서 서버가 직접 구글/애플 스토어 API와 통신하여, 클라이언트가 보낸 영수증 토큰이 유효한지 2차로 교차 검증(Validation)해야 재화 어뷰징을 막을 수 있습니다.
