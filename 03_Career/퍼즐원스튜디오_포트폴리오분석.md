# 🎯 퍼즐원스튜디오 포트폴리오 합격 포인트

이 포트폴리오는 단순한 '게임 제작기'가 아니라, 엔진의 한계를 수학과 알고리즘으로 직접 돌파한 '**엔지니어링 리포트**'다. 실제 기술 면접의 질문 80% 이상이 이 포트폴리오의 트러블슈팅 페이지에서 파생되었다.

## 1. 엔진 비의존성과 수학/물리 하드코딩 (Drawing Roller)
모바일 캐주얼 게임사에서 가장 사랑하는 '성능과 조작감을 수학으로 해결하는 역량'을 어필한 프로젝트. 면접관의 눈길을 가장 먼저 사로잡은 핵심 무기였다.

> 📄 **포트폴리오 6~7페이지: 커스텀 탄성 물리 및 Verlet 적분법**
> ![[퍼즐원스튜디오_포트폴리오.pdf#page=7]]>  ![[퍼즐원스튜디오_포트폴리오.pdf#page=8]]
* **🎯 합격 포인트:** 무거운 내장 물리 엔진(Rigidbody) 없이 Hooke's Law와 Verlet 적분법을 코드로 직접 구현해 물리 시뮬레이션을 완성함.
* **🗣️ 실제 면접 질문:** "Verlet 적분법에서 코드로 옮기는 게 어려웠는지? 실제 물리 연산과 차이가 많이 나는지?"
* **💡 나의 방어:** "코드 구현 자체보다는 물리 법칙을 이해하는 게 더 오래 걸렸다. 오차는 미미했으나 시각적 안정을 위해 후보정(Constraint)이 필요했다."

> 📄 **포트폴리오 5페이지: Stencil Masking 렌더링**
> ![[퍼즐원스튜디오_포트폴리오.pdf#page=6]]
* **🎯 합격 포인트:** 연산 비용이 높은 Mesh Cutting이나 Alpha Masking 대신, Stencil Buffer를 직접 제어하는 렌더링 파이프라인 구축.
* **🗣️ 실제 면접 질문:** "Stencil buffer로 최적화 어떻게 했는지, 실제 측정한 사례가 있는지?"
* **💡 나의 방어:** "이론적으로 알파 블렌딩보다 부하가 적어 도입했다. 다만 당시 물리 연산 부하가 더 심각해 정확한 수치 측정은 후순위로 미뤘다." (솔직한 프로파일링 우선순위 어필)

## 2. 프로파일링 기반의 병목 해결 (Tiny Survivor)
클라이언트 개발자임에도 서버를 뜯어보며 대규모 객체의 '최적화' 원리를 규명한 프로젝트.

> 📄 **포트폴리오 13페이지: 프로파일링을 통한 A* 알고리즘 병목 탐지**
> ![[퍼즐원스튜디오_포트폴리오.pdf#page=14]]
* **🎯 합격 포인트:** 프로파일링 툴을 이용해 CPU 점유율의 10.92%를 차지하던 A* 알고리즘 병목을 찾아내고, 이를 경량화된 추적 알고리즘으로 전환하여 점유율을 0.92%로 낮춤.
* **🗣️ 실제 면접 질문:** "Astar와 추적 알고리즘 변경 이유와 장단점?"
* **💡 나의 방어:** "수십 마리의 몬스터가 일괄 탐색하니 부하가 심했다. 게임 특성상 미로 같은 복잡한 구조가 아니었기에 단순 추적 알고리즘만으로도 충분했다." (오버 엔지니어링 방지 및 기획적 타협 어필)

## 3. 알고리즘 응용과 데이터 파이프라인 (Memoria Labyrinth)
절차적 생성이라는 불확실성을 '그래프 이론'이라는 수학적 근거로 완벽하게 통제해 낸 사례.

> 📄 **포트폴리오 21페이지: 절차적 던전 생성 (MST)**
> ![[퍼즐원스튜디오_포트폴리오.pdf#page=22]]
* **🎯 합격 포인트:** 무작위로 방을 생성할 때 발생하는 '고립된 방' 문제를 해결하기 위해, 그래프 이론의 MST(Kruskal 알고리즘)를 적용해 100% 연결성을 수학적으로 보장함.
* **🗣️ 실제 면접 질문:** "절차적 생성 알고리즘 설명?"
* **💡 나의 방어:** "기본 베이스 룸에서 인접한 곳에 방을 생성하고, MST를 이용해 모두 연결되도록 구현했다."

## 4. 메모리 관리 및 프레임 방어 (화재 대비 시뮬레이터)
잦은 객체 생성/파괴가 일어나는 환경에서 메모리 단편화와 프레임 드랍을 어떻게 방어하는지 보여준 핵심 프로젝트.

> 📄 **포트폴리오 28페이지: Replicated Object Pooling**
> ![[퍼즐원스튜디오_포트폴리오.pdf#page=29]]
* **🎯 합격 포인트:** 화재 이펙트처럼 생성과 파괴가 잦은 객체들의 메모리 할당/해제 비용을 없애기 위해 Object Pooling 시스템을 구축함.
* **🗣️ 실제 면접 질문:** "오브젝트 풀링에서 Stack을 사용한 이유와 Queue와의 차이점? 순간 Spike 해결법은?"
* **💡 나의 방어:** "방금 해제된 메모리를 재사용하는 캐싱 룰을 기반으로 Stack을 썼다. 스파이크 방지를 위해 Capacity를 점진적으로 크게 늘리거나 코루틴을 이용해 분산 처리했다."
